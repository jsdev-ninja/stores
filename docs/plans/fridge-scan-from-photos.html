Build My Order From Fridge Photos

Customer photographs their fridge; Gemini identifies products; the system diffs against a saved baseline and auto-builds a reviewable cart.

TypeFeature
Date2026-06-17
Project@jsdev-store (storebrix.com)
StackFirebase Functions v2 · Genkit/Vertex Gemini 2.5 · React 19 + Vite · HeroUI v3 · Algolia
Planned bysoftware-architect

Goal

Let a returning grocery customer rebuild their weekly order in seconds: photograph the fridge/pantry, have AI detect what is present, compare it to a personal "full fridge" baseline, and present a pre-filled, fully editable cart of what is missing — turning a 30-minute re-shop into a 2-minute photo-and-approve.

Scope

In scope
  • New backend module functions/src/modules/fridgeScan/ with two callables: analyze-photos and confirm-template.
  • Gemini vision flow (Genkit/Vertex, mirroring chatbot wiring) that returns Hebrew product+brand names and per-item quantity + confidence.
  • Template-constrained Algolia matching: when a template exists, rank detections against the user's ~30 template SKUs first, falling back to the full tenant index.
  • Pure diff/quantity engine: order_qty = max(0, template_qty − detected_qty).
  • InventoryTemplate core entity (with source: "photo" | "history" | "manual") + pluggable seeding-service boundary.
  • Per-scan log collection for cost/usage visibility.
  • Frontend: new route store.fridgeScan, first-run setup wizard, returning-scan screen, cart-review screen, Redux domain + appApi methods.
  • Confidence policy wired into review UX (auto-accept / confirm / drop).
  • Cost & abuse controls: per-user rate limit on analyze, hard image cap (max 6), client + server image downscaling, structured per-scan cost logging.
  • Recurring-scan / restock reminder (scheduled job + notification), as a final shippable increment.
  • Coordinated @jsdev_ninja/core version bump (0.17.0 → 0.18.0) propagated to both consumers.
Out of scope
  • Seed-from-order-history baseline (designed as a drop-in boundary; not built — see Future Enhancement).
  • Any change to checkout, payment, ledger, or order placement — the feature ends when items land in the existing cart.
  • Editing the Algolia index schema or re-indexing products.
  • New external service or new secret — reuses existing Vertex/Genkit credentials.
  • Barcode scanning / OCR of receipts (photo-only vision).
  • Multi-template per user (one active template per {companyId}/{storeId}/userId for v1).
  • Admin UI for browsing customer templates/scans (logging is backend-only for v1).
  • Money math beyond what the existing cart already computes (the diff engine is quantity-only).

User Flows

Flow A — First-run SETUP (build the baseline template)

  1. UserOpens "Build order from fridge" entry point; has no template yet, so the app routes into the setup wizard.
  2. UserStep 1 — captures/uploads 1–6 photos of the full fridge/pantry/freezer (camera or file). UI enforces the cap and shows thumbnails.
  3. SystemClient downscales each image (longest edge ≤ 1280px, JPEG q≈0.7) and uploads to Storage under {companyId}/{storeId}/fridge-scans/{scanId}/{n}.jpg.
  4. BackendanalyzeFridgePhotos callable: rate-limit check → server-side downscale guard → Gemini vision detect → Algolia match (no template yet → full tenant index) → returns detected items with matched product, qty, confidence, and a confidence band.
  5. UserStep 2 — confirm/correct screen: auto-accepted items (conf > 0.90) pre-checked; medium items (0.60–0.90) flagged "please confirm"; low items already dropped. User can add missing products via search, edit quantities, or remove.
  6. UserStep 3 — taps "Save as my full fridge".
  7. BackendconfirmInventoryTemplate callable persists the corrected list as the active InventoryTemplate (source: "photo") with productId + fullQty per line.
  8. SystemWizard completes; user is offered "scan now to build an order" (jumps straight into Flow B).

Flow B — Returning SCAN (build the order)

  1. UserOpens the entry point; an active template exists, so the app shows the scan screen directly.
  2. UserCaptures/uploads 1–6 photos of the current fridge.
  3. SystemClient downscales + uploads to Storage (same path convention, new scanId).
  4. BackendanalyzeFridgePhotos (mode: "scan"): Gemini detect → template-constrained Algolia match (restrict ranking to the user's template SKUs first) → diff engine computes orderQty = max(0, fullQty − detectedQty) per template line → returns review rows.
  5. UserCart-review screen: one row per template item showing "have N / need M", with auto/confirm/low styling. Everything editable. Items the user has "enough" of show qty 0 (excluded by default).
  6. UserTaps "Add to cart" (or "Add to cart & checkout").
  7. SystemFrontend writes the reviewed lines into the existing cart in a single bulk setV2<TCart> write (merging with existing items), then navigates to the cart page.
  8. UserReviews the normal cart and proceeds through the existing checkout. Feature ends here.
Non-DB persistence — Firebase Storage Uploaded photos live at {companyId}/{storeId}/fridge-scans/{scanId}/{index}.jpg (tenant-scoped, mirrors existing {companyId}/{storeId}/... convention). Photos are passed to Gemini by signed/download URL or as inline base64 from the just-uploaded blobs. A scheduled cleanup (or Storage lifecycle rule) deletes scan images older than 7 days — raw fridge photos are PII-adjacent and must not accumulate.

Key design choice — diff and matching happen on the backend, cart write happens on the client. The callable returns review rows only; it never mutates the cart. This keeps the callable side-effect-light (idempotent-friendly, cheap to retry) and reuses the existing client cart-write path verbatim, avoiding a second source of truth for cart mutation.

Backend Module Design

Module name: fridgeScan (camelCase, matching existing chatbot/catalog convention). Justification: the feature is a cohesive bounded context (vision detection + template + diff) that owns its own Firestore paths and emits its own events; it does not belong inside cart (which it never writes from the backend) or catalog (which it only reads via Algolia).

Both are functionsV2.https.onCall with invoker: "public", memory: "1GiB", timeoutSeconds: 540 (mirroring chatbotApi — vision calls are slow). Both re-exported from functions/src/index.tsx via export { analyzeFridgePhotos, confirmInventoryTemplate, restockReminderSchedule } from "./modules/fridgeScan". Full Zod in the Contracts appendix.

  • confirmInventoryTemplate(ConfirmTemplateRequest) → ConfirmTemplateResponse — persists the active template. Idempotent on idempotencyKey.

Vision Detection & Matching

Genkit vision flow (detectProductsFromImages.ts)

A single ai.generate call with multimodal input: the system/Hebrew prompt plus each image as a media part ({ media: { url } } for Storage download URLs, or inline base64). Output is forced to a Zod array via output: { schema: DetectedItemArraySchema } so detection returns structured rows, not prose. No tools are needed for detection (unlike the chatbot) — this is a single structured-extraction call.

Hebrew detection prompt strategy

The Algolia products index stores Hebrew-only names (name: {lang:"he", value}[]) and a Latin/Hebrew brand string. To maximize match rate the model must emit Hebrew product nouns and Israeli brand names. Prompt requirements (English instructions, Hebrew outputs):

  • Instruct: "You are a grocery-inventory vision assistant for an Israeli supermarket. Identify each distinct food/household product visible across the images."
  • "Return the product name and brand in Hebrew (the catalog is Hebrew). Use the on-package Hebrew text when legible."
  • Provide Israeli brand exemplars so the model anchors to local brands: תנובה, טרה, אסם, שטראוס, עלית, פרי הגליל, תלמה, יטבתה, מולר, יופלה (Tnuva, Tara, Osem, Strauss, Elite, Pri HaGalil, Telma, Yotvata, Muller, Yoplait).
  • For each item return: heName (e.g. חלב 3%), heBrand (nullable), quantity (integer count of distinct units visible), unitHint ("unit" | "kg" | "liter" | "ml" | "gram" | "unknown"), and rawConfidence 0–1.
  • Counting guidance: "Count only clearly visible distinct units. If items are stacked/occluded and you cannot count, set quantity to your best minimum and countUncertain: true." (Feeds the low/absent fallback — see Risks.)
  • "Do not invent products you cannot see. Omit anything you are unsure exists."

Matching algorithm (matchDetectionsToProducts.ts)

Constraint discovered in code: the existing AlgoliaService.queryProducts(query) takes only a query string and hardcodes filters: storeId:X AND companyId:Y internally; it returns full TProduct[] hits but does not surface a relevance score, and offers no way to constrain to a SKU set. The matching service therefore wraps Algolia rather than calling queryProducts directly, so it can (a) request the Algolia _rankingInfo/score and (b) pass filters that AND-in the template SKUs. The tenant filter (storeId AND companyId) MUST be preserved on every call.
  • Template-constrained pass (scan mode, when a template exists): for each detection, search Algolia with query = "{heBrand} {heName}" and filters: storeId:X AND companyId:Y AND (sku:S1 OR sku:S2 OR ...) built from the template's SKUs. This narrows ~3,600 products → the ~30 the user buys — the primary accuracy unlock.
  • Fallback pass (setup mode, or no template hit): search the full tenant index (filters: storeId:X AND companyId:Y only).
  • Confidence scoring: matchConfidence = f(rawConfidence_from_Gemini, algoliaTopScore, scoreGapToSecondHit). Concretely: start from rawConfidence; boost when the top Algolia hit's normalized score is high and well-separated from the runner-up; penalize when the top two hits are close (ambiguous) or the brand mismatches. Clamp to [0,1].
  • Confidence policy (per spec, applied to matchConfidence): > 0.90 → band: "auto" (pre-accepted); 0.60–0.90 → band: "confirm"; < 0.60 → band: "drop" (returned but excluded from defaults, surfaced under "we weren't sure").
  • No match found: detection is returned with product: null, band: "drop" so the UI can still show "we saw X but couldn't match it".

Pure diff engine (computeOrderDiff.ts)

Zero I/O, fully unit-testable. Input: the active InventoryTemplate + matched detections. For each template line: find detections matched to that productId, sum their quantity → detectedQty; orderQty = max(0, fullQty − detectedQty). Output one ReviewRow per template line (plus "extra" rows for confident detections not in the template, offered as optional adds in setup). Money is never touched here — see the Money note below.

Money note (verified in code): despite the agorot rule in CLAUDE.md, the live cart treats product.price as shekels (cart-store.ts selectCost does product.price * CONFIG.VAT/100 and renders via Intl.NumberFormat('he-IL', ILS)). This feature does no money math — the diff engine outputs quantities only, and any price shown on the review screen must reuse the existing Price component / formatter.price path (which expects shekels). Do not multiply or divide by 100 anywhere.

The page decides setup-vs-scan based on whether an active template exists.

  • SetupWizard.tsx — 3 sections driven by local state ("capture" | "review" | "done"), like CheckoutPage. Hosts the photo step, the confirm/correct grid, and the save action.
  • PhotoCaptureStep.tsx — multi-photo capture (max 6, enforced). Reuse/extend components/FIleUpload/FileUpload.tsx — note it is currently single-file; either render N instances or extend it to accept multiple. Each selected file is downscaled (canvas) then uploaded via firebaseStorage.upload(path, file).
  • DetectionReviewGrid.tsx — shared by setup + scan. One row per item: product name/image, qty stepper, confidence styling (auto = subtle check, confirm = amber "אשרו" prompt, drop = greyed under a "לא היינו בטוחים" group). Product search-to-add reuses the catalog Algolia widget scope or a simple lookup.
  • ScanScreen.tsx — capture step → calls analyze → renders CartReviewScreen.
  • CartReviewScreen.tsx — "have N / need M" rows from the diff; editable; "Add to cart" / "Add & checkout" actions.

Redux domain + appApi methods

  • New domain apps/store/src/domains/fridgeScan/ (slice + types), holding transient scan state (status, reviewRows, scanId, error) and the cached active template. Mirrors existing domains.
  • appApi additions (in apps/store/src/appApi/index.ts, user namespace): fridgeScan.analyze({ scanId, imageRefs, mode }) and fridgeScan.saveTemplate({ items, idempotencyKey }) — thin wrappers over httpsCallable(functions, "analyzeFridgePhotos") / "confirmInventoryTemplate" (same pattern as chatbotChat in lib/firebase/api.ts); plus fridgeScan.getActiveTemplate() reading inventory-templates/{userId} via FirebaseApi.firestore.getV2 + getPath.

Loading reviewed items into the cart — RECOMMENDATION

Use a single bulk setV2<TCart> write, NOT an addItemToCart loop. addItemToCart reads-modifies-writes the whole cart doc once per product and only increments by 1 — looping it would be N racy full-doc writes and can't set arbitrary quantities. The reviewed list maps cleanly to TCartItemProduct[] ({ product, amount }), so build the merged items array (existing cart items + reviewed rows, summing amounts on product.id collision) and write it once via FirebaseApi.firestore.setV2<TCart>({ collection: getPath({...,collectionName:"cart"}), doc: { ...actualCart, items } }) — exactly the shape createCartFromOrder already uses. One write, correct quantities, no races.

i18n / RTL (Hebrew primary)

  • Add a fridgeScan.* i18next namespace. Keys (Hebrew values): title (בנו הזמנה מתמונות המקרר), setup.cta, capture.hint, review.have / review.need, confidence.confirm (אשרו פריט זה), confidence.unsure (לא היינו בטוחים), saveTemplate (שמרו כמקרר מלא), addToCart, error.rateLimited (נסו שוב בעוד מספר דקות), error.tooManyImages.
  • Layout inherits RTL from document.body.dir; steppers/grids must be RTL-safe (no hard-coded left/right).

Cost & Abuse Controls (mandatory — all-stores, no flag)

Because there is no feature flag or gating, these are first-class requirements, not nice-to-haves. Every one is enforced server-side; client-side measures are UX only and must never be trusted as the limit.
  • Per-user rate limit (server): checkScanRateLimit.ts counts the user's fridge-scans docs in a sliding window (e.g. ≤ 5 scans / hour and ≤ 20 / day per userId). Over limit → HttpsError("resource-exhausted"). Window is read from the log collection (already written per scan), so no extra store needed.
  • Hard image cap (server + client): server rejects imageRefs.length > 6 with invalid-argument; client disables capture past 6. Cap is the dominant cost lever (Gemini bills per image).
  • Image downscaling (client + server guard): client downscales to longest-edge ≤ 1280px, JPEG q≈0.7 before upload (canvas). Server validates each image's byte size is under a ceiling (e.g. 1.5 MB) and rejects oversized payloads rather than forwarding them to Gemini.
  • Structured per-scan cost logging: every analyze writes a fridge-scans doc with imageCount, model, token usage, and estCost (agorot), and a logger.info("fridgeScan.analyzed", {...}) structured line (no images/PII). This is the cost-visibility surface.
  • Metric to watch: daily sum of fridge-scans.estCost per store + scans-per-user p95. A spike in either is the early-warning signal for abuse or a prompt regression. (Surface later via the existing analytics/Mixpanel path or a simple admin query — admin UI is out of scope for v1.)
  • Quota-error handling: mirror chatbotApi — catch 429/quota errors and return a friendly "temporarily unavailable" message instead of a hard throw.

Recurring-scan / Restock Reminders

Final shippable increment. Reuses the verified scheduled-trigger + event-bus + notification patterns already in the repo (onSchedule in budget/triggers/reconcileProjectionsSchedule.ts; subscribe(...) + emailService in notifications/subscribers/orderPlacedAdminEmail.tsx).

  • triggers/restockReminderSchedule.ts — onSchedule({ schedule: "every day 09:00", timeZone: "Asia/Jerusalem" }). Iterates stores (like the budget job), then templates where remindEveryDays is non-null and now − lastScanAt ≥ remindEveryDays·86400000. For each due template, emitEvent a restock-due event (no PII beyond ids/email).
  • subscribers/sendRestockReminderOnDue.ts — subscribe to restock-due → send a notification (email via existing emailService, and/or FCM via the existing messaging path) with a deep link to /fridge-scan. Idempotent via deterministic doc id evt_sendRestockReminder_{eventId}.
  • Sentinel handling: remindEveryDays = null means reminders OFF — the scheduler MUST skip null/≤0 before the date math (documented in Security Invariants).
  • UI: a simple "remind me every N days" toggle on the template (writes remindEveryDays). Defaults to off.

Future enhancement — seed-from-order-history (designed, not built)

The seedTemplate.ts service is the single seam. v1 implements only the { source: "photo", items } path. A future history path adds a { source: "history" } branch that reads the user's recent orders, aggregates typical quantities per product, and returns the same TInventoryTemplateItem[] shape — no change to the entity, the diff engine, the callables, or the UI review grid. Because source already exists on the entity and the seeding boundary is isolated, this is additive.

Security Invariants

Authentication

Both callables require a signed-in, non-anonymous user. Derive userId from auth.uid; reject (or no-op) anonymous/absent auth. Mirror chatbotApi's auth handling but stricter — cart/template writes must not happen for anonymous users.

Authorization (tenant + owner isolation)

companyId/storeId come from auth.token (fallback to verified store context exactly as chatbotApi does), NEVER from raw client input. The template is keyed by userId and read/written only at {companyId}/{storeId}/inventory-templates/{userId} — a user can only ever read/write their own template and their own scan logs. No cross-user or cross-tenant read path exists.

Input validation & trust boundary

Validate every request with the Zod contracts. imageRefs max 6 and each path MUST be under the caller's own {companyId}/{storeId}/fridge-scans/ prefix (reject foreign Storage paths — prevents pulling another tenant's images into Gemini). confirmInventoryTemplate re-fetches each product by id tenant-scoped to fill sku/name/image; client-sent denormalized fields are ignored. fullQty must be a positive integer.

Data boundaries & PII

Raw fridge photos are PII-adjacent: store ONLY under the tenant-scoped Storage path, never log image bytes/URLs or detection text, and delete after 7 days. The fridge-scans log holds counts/model/cost only. Structured logs use firebase-functions/v2 logger with ids only.

Trust escalation — recompute server-side

The cart is written by the client (reusing the existing path), so do NOT trust client-claimed prices: the cart write must embed the full TProduct read from the store/Algolia (as today), not a client-fabricated product. orderQty is advisory; the authoritative cart total is recomputed by the existing cart/checkout/ledger path unchanged.

Sentinel / magic values

remindEveryDays = null means reminders OFF. The scheduler MUST guard null and ≤ 0 before any date arithmetic, or it would spam every user. Documented and tested.

Abuse / cost (no feature flag)

Per-user rate limit (resource-exhausted on exceed), hard image cap, and server-side image-size ceiling are security controls here, not just UX — there is no gating to contain a runaway. Quota/429 from Vertex returns a friendly message, never an unhandled throw.

External services / secrets

No NEW external service and no new secret — reuses the existing Vertex AI / Genkit credentials already used by the chatbot. If the project later moves Gemini auth to a dedicated key, prefer defineSecret/Secret Manager over plaintext .env (current pattern is plaintext); but nothing new is required for this feature.

Breaking Changes

None — all additive. New core entities, two new collections registered in storeCollections, new callables, new route, new Redux domain. The core version bump (0.17.0 → 0.18.0) is additive (no existing entity changes), and both consumers are updated in the same coordinated step (Backend Task B1), so no consumer is left on a stale version.

Risks & Open Questions

High

Quantity counting reliability (occlusion / stacking). The baseline helps identification (narrowing 3,600→30 SKUs) but does NOT make counting easier — a fridge with milk behind milk is genuinely hard to count from a photo. Mitigations: (1) template-constrained identification so at least "is it present?" is reliable; (2) a coarse low/present/absent signal as a fallback when countUncertain is true — treat "uncertain + present" as "you may have some, qty defaults conservative"; (3) the review screen is ALWAYS fully editable so the human is the final counter; (4) frame the UX as "we pre-filled your usual order, adjust quantities" rather than promising exact counts. The vision spike (Step 0) decides how far to lean on exact counts vs. the coarse fallback.

High

Hebrew identification accuracy for Israeli brands. Match rate depends on Gemini emitting Hebrew names/brands that Algolia can match against Hebrew-only product names. Mitigations: Hebrew-forcing prompt with Israeli brand exemplars (Tnuva/Tara/Osem/…), template-SKU-constrained Algolia search, and the confidence bands so low-confidence matches never auto-apply. Measured in the spike.

Medium

Per-scan cost & abuse at all-stores scale, no flag. Each scan = up to 6 images through Gemini. Mitigations are first-class: image cap, client+server downscale, per-user rate limit, per-scan cost logging + a daily-cost metric. Residual risk if a store has many power users — the metric is the early-warning.

Medium

Algolia score exposure for confidence scoring. The existing service doesn't surface relevance scores; the matching service must request _rankingInfo/score directly from the index. If score isn't reliably comparable across queries, confidence may need to lean more on the Gemini rawConfidence + brand-string agreement. Validate during B5.

Low

Multimodal input shape in this Genkit version. Gemini 2.5 via Vertex supports image parts, but the exact media-part shape (download URL vs. inline base64, gs:// support) should be confirmed against genkit 1.30.1 in the spike. Low risk — well-trodden path.

Low

Cart id is z.string().uuid(). When building the cart write, reuse the existing actualCart doc / generateDocId("cart") path (as createCartFromOrder does) so ids stay valid; don't fabricate non-UUID ids. The InventoryTemplate's own id is also a UUID, but its Firestore doc id is userId (intentional, for one-per-user idempotency).

Open

Confirm rate-limit thresholds (proposed: 5/hour, 20/day per user) and the image cap value within 4–8 (proposed: 6). Acceptable?

Open

One active template per user for v1 (no named multi-templates "kitchen" vs "garage")? Plan assumes yes.

Open

Reminder channel: email only (existing emailService), push (FCM via existing messaging), or both? Plan assumes email + optional push.

Open

Photo retention: 7 days proposed. Acceptable for privacy, or should photos be deleted immediately after the scan completes (we keep only the log)? Immediate-delete is more privacy-friendly and cheaper; 7 days helps debugging.

Open

Entry-point placement: cart page button, catalog header, and/or a dedicated nav item? Plan assumes a cart/catalog-header button for v1.
