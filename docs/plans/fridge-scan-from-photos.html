Storage-Place Inventory From Photos (B2B)

An organization (office) sets up each storage place — fridge, cupboard, freezer — by photographing it from several angles; Gemini merges the angles into one estimated inventory the user confirms as a long-term baseline. Later: re-photo a place to see what is missing and reorder.

What Changed Since the Last Revision

This is now a B2B / organization feature, not a consumer one. The previous revision modelled a single consumer scanning their home fridge with one per-user InventoryTemplate. That model is removed. An organization (office) owns multiple storage places (fridge / cupboard / freezer / shelf), each set up independently and shared across all of the org's users.
  • Entity replaced: per-user InventoryTemplate → org-scoped StoragePlace. The baseline now lives on the place, not on a user.
  • Scope changed from per-user to per-organization. Any member of the org sees and edits the same set of places (grounded in the existing organizationId field + profile.organizationIds membership model — see Org Model section).
  • Multi-angle reconciliation is now core. Up to 5 photos per place go in ONE Gemini call that de-dups the same physical item across angles and uses angles to see behind items. Isolated behind a reconcilePlaceInventory boundary so the strategy can be swapped.
  • Baseline photos are kept long-term as the reference image set for the place (the previous "delete scan photos after 7 days" rule applies only to the later diff/what's-missing flow, not to baseline photos).
  • Counts are explicitly best-effort estimates the user confirms — never promised as exact. This is now backed by spike evidence (see Spike section).
  • Auth/infra corrected: detection uses the Gemini Developer API via googleAI({ apiKey: process.env.GOOGLE_GENAI_API_KEY }) — NOT Vertex, NOT defineSecret. This matches the code already in the module. (The previous revision said Vertex; that was wrong.)
  • A spike happened. Identification of real Israeli brands and separated-item counting are validated; single-image depth counting is not. Findings are captured in their own section and drive the design.

Goal

Let an office stop hand-counting supplies: photograph each storage place once to capture an estimated "full" baseline the team confirms, then later re-photograph a place to instantly see what has been used up and rebuild the restock order — turning a manual stock-take into a photo-and-confirm.

Scope

In scope (now)
  • New org-scoped StoragePlace entity in @jsdev_ninja/core (replaces InventoryTemplate), tenant-scoped under the store with an organizationId field.
  • Backend module functions/src/modules/fridgeScan/ grows from a detection-only spike into: createStoragePlace, listStoragePlaces, analyzePlacePhotos (multi-angle merge), confirmPlaceBaseline.
  • Reconciliation Approach #1 behind a reconcilePlaceInventory boundary: all photos → ONE Gemini call → ONE merged, de-duped inventory with per-item count + uncertainty flag.
  • Depth-aware prompt: count behind an item only with visible evidence (cap/edge), flag depthUncertain, lean on multi-angle as the real depth signal.
  • Setup UI for org users: a "Storage places" area to create & name places, pick a type, upload up to 5 angle photos, review the merged draft inventory, adjust counts, and confirm the baseline.
  • Baseline photos stored long-term in tenant-scoped Storage and referenced on the place.
  • Per-analyze cost/usage log (counts, model, tokens, est cost) — no images/PII in the log.
  • Coordinated @jsdev_ninja/core version bump (0.21.0 → 0.22.0) propagated to apps/store and functions.
Out of scope (later / not now)
  • The "what's missing" diff flow — re-photo → missing = baseline.count − current.count → reorder list. Designed here, built in a later increment (reuses the same place + merge engine).
  • SKU matching to a real cart — turning a baseline label + count into an Algolia store SKU is UNTESTED and is the next spike. Without it there is no cart. Capture label + count now; no cart in this scope.
  • Any change to checkout, payment, ledger, or order placement.
  • Editing the Algolia index schema or re-indexing products.
  • Per-floor hierarchy / nesting of places (a flat list of places per org for v1; type distinguishes them).
  • Admin UI for browsing org baselines/scan logs (logging is backend-only for v1).
  • Restock reminders / scheduled re-scan prompts (revisit after the diff flow exists).

User Flows

Flow A — SET UP a storage place (build the baseline) — IN SCOPE

  1. UserAn org member opens "Storage places" (visible only when an active organization is selected). Sees the org's existing places (empty on first use) and an "Add place" action.
  2. UserCreates a place: enters a name (e.g. מקרר קומה 2) and picks a type (fridge | cupboard | freezer | shelf | other). createStoragePlace persists an empty place (baseline.status = "draft").
  3. UserUploads up to 5 photos of the place from different angles. UI enforces the cap and shows thumbnails.
  4. SystemClient downscales each image (longest edge ≤ 1280px, JPEG q≈0.7) and uploads to {companyId}/{storeId}/storage-place-photos/{placeId}/{n}.jpg (kept long-term).
  5. BackendanalyzePlacePhotos: verifies org membership → passes ALL photos to reconcilePlaceInventory (ONE Gemini call) → returns ONE merged draft inventory: "5 cola, 3 bread, 4 milk…", each row with count, unit, confidence, and a depthUncertain flag.
  6. UserReviews the merged draft: adjusts counts (e.g. 5 → 4), adds items the model missed, removes false positives. Uncertain rows are visually flagged "please confirm".
  7. UserTaps "Save as full" → confirmPlaceBaseline persists the confirmed items + the photo refs as the place's baseline (baseline.status = "confirmed").
  8. UserRepeats per place. The org now has a confirmed baseline per place.

Flow B — "What's missing" diff & reorder — DESIGNED, LATER

  1. UserOpens a place that already has a confirmed baseline; taps "Check what's missing".
  2. UserRe-photographs the place (up to 5 angles); client downscales + uploads as a short-lived scan (deleted after the diff, unlike baseline photos).
  3. BackendanalyzePlacePhotos (same engine) returns the current merged inventory.
  4. SystemPer baseline item: missing = max(0, baseline.count − current.count) → a "needs restock" list.
  5. UserReviews/edits the restock list. (Turning this into a cart requires the SKU-matching spike — out of scope.)
Non-DB persistence — Firebase Storage (two retention classes) Baseline photos live at {companyId}/{storeId}/storage-place-photos/{placeId}/{n}.jpg and are kept long-term (they ARE the reference for the place; their refs are stored on baseline.photos). Diff-scan photos (Flow B, later) are short-lived and deleted after the diff completes. Both are tenant-scoped and never logged.

Key design choice — the merge is isolated behind reconcilePlaceInventory. Approach #1 (all photos → one Gemini call → one merged inventory) is the only strategy implemented now, but the callable and the orchestrator depend on the boundary's signature, not on Gemini. A later strategy (per-image detect → server-side reconcile, or a SKU-aware pass) drops in without touching the callables, the entity, or the UI.

Spike — Validated / Findings

A detection spike ran against the real detectProductsFromImages flow (Gemini Developer API, gemini-2.5-flash) on 4 real Israeli fridge photos plus generated images. These findings are load-bearing — they justify the "estimate the user confirms" framing and the multi-angle requirement.

PASS

Identification of real Israeli brands off real labels works. Correctly read Tnuva, Carmel / כרמל, Tomer / תומר, Danone, Tara, Maya, and Coca-Cola from real packaging. This is the core unlock — the model can name local products in context.

PASS

Counting clearly-separated items is reliable. visibleCount nailed milk cartons (1/1/2/3 across photos) and eggs (6). When units are visually distinct and not occluding each other, counts are trustworthy.

WARN

Single-image depth / occlusion counting is unreliable. It over-counts when nothing is behind (read milk ×2 with only one carton) and under-counts real depth (4 cola bottles read as 2–3). The depthUncertain flag fired inconsistently. Mitigation: multi-angle photos are the real depth signal, and every count is human-confirmed before it becomes a baseline.

PASS

Bonus — detects EMPTY containers. Flagged an empty cola bottle. Useful for restock signals (an empty unit = needs reordering even though it's "present").

FAIL

Generated images are useless for brand ID. Fabricated labels make brand identification meaningless. Generated images are only useful for stress-testing variety / clutter / counting — never for validating identification accuracy.

PASS

Cost per call is tiny. Not a constraint at expected volumes; cost logging is for visibility, not a hard limiter.

Design consequences of the spike (non-negotiable): (1) counts are presented as estimates the user confirms, never as exact; (2) multi-angle (up to 5) is required precisely because single-image depth fails; (3) the review step is always fully editable — the human is the final counter; (4) depthUncertain is surfaced in the UI but treated as advisory, not authoritative.

Organization Model (verified in code)

The data model and UI location below are grounded in what already exists — nothing here invents an org concept.

  • Organization entity exists: packages/core/lib/entities/Organization.ts — TOrganization { id, name, billingAccounts[], paymentType, branches?[], … }. Stored at {companyId}/{storeId}/organizations/{organizationId} ("organizations" is already a registered storeCollections key).
  • Membership: a user's TProfile (packages/core/lib/entities/Profile.ts) carries organizationIds: string[] (current) and organizationId (deprecated single value). A user can belong to multiple orgs.
  • Active org (client): Redux holds the selected org at state.userOrganization.activeOrganization (TOrganization | null). OrgPicker switches it; CheckoutPage reads it as profileOrganization to stamp order.organizationId.
  • Existing org-scoping pattern: org data lives at the normal tenant path and is filtered by an organizationId field — e.g. organizationBalance, budgetRecords, orders all use .where("organizationId", "==", id). StoragePlace follows this exact pattern (a field + a where-clause), NOT a nested sub-collection path.
  • Backend authorization reality (important): the one existing org-scoped callable (documents/getOrganizationBalance) is admin-only — it takes companyId/storeId from the token and trusts the organizationId in the request because only admins call it. There is no existing pattern for a non-admin org member proving they own an organizationId. This feature is used by ordinary org members, so it MUST add that check (read caller's profiles/{auth.uid} and assert organizationIds.includes(organizationId) — see Security Invariants). The profile-read-then-verify shape is modelled on customers/deleteClient.ts.

Callable signatures

All four are functionsV2.https.onCall with invoker: "public"; analyzePlacePhotos uses memory: "1GiB", timeoutSeconds: 540 (vision is slow) — the other three can be smaller (256MiB). All re-exported from functions/src/index.tsx. Full Zod in Contracts.

  • createStoragePlace(CreateStoragePlaceRequest) → { placeId } — creates an empty place for the caller's org.
  • listStoragePlaces(ListStoragePlacesRequest) → { places: TStoragePlace[] } — lists the org's places.
  • analyzePlacePhotos(AnalyzePlacePhotosRequest) → { placeId, items: TReconciledItem[], warning } — multi-angle merge; pure read (creates only a log doc). Never writes the baseline.
  • confirmPlaceBaseline(ConfirmPlaceBaselineRequest) → { placeId, itemCount } — writes the confirmed baseline. Idempotent on idempotencyKey.

Vision Detection & Reconciliation

Reconciliation — Approach #1 (reconcilePlaceInventory.ts)

ONE ai.generate call with multimodal input: the depth-aware system prompt plus every uploaded image as a media part. The model is instructed to treat the images as different angles of the same place and return ONE merged inventory — de-duplicating the same physical item seen from multiple angles, and using angles to resolve what is behind what. Output forced to a Zod array via output: { schema: ReconciledItemArraySchema }.

Why this is a boundary, not just a function. The merge logic ("same item across angles") lives entirely inside this file. Approach #1 puts the merge inside the single Gemini prompt. A future Approach #2 (per-image detect, then server-side reconcile with embeddings/heuristics) implements the same reconcilePlaceInventory(args) → TReconciledItem[] signature and swaps in with zero changes upstream.

Depth-aware Hebrew prompt strategy

  • "You are a grocery-inventory vision assistant for an Israeli office. The images are different angles of the same storage place — do not double-count an item seen from two angles."
  • Identify each distinct product; return label and brand in Hebrew using on-package text. Anchor to Israeli brands (Tnuva / תנובה, Tara / טרה, Osem / אסם, Strauss, Elite, Carmel / כרמל, Tomer / תומר, Danone, Maya, Coca-Cola).
  • Depth rule (from spike): "Count an item behind another ONLY when you can see direct visual evidence of it (a visible cap, edge, or label of a second unit). Do not assume hidden units. If you suspect more behind but cannot confirm, set depthUncertain: true and report the count you can actually see." Multi-angle is the depth signal — items visible only in one angle still count once.
  • Partial-item rule: "A partially-visible item behind an identified one is assumed to be the SAME product at lower confidence, unless it is clearly a different product."
  • Per item return: label (Hebrew), brand (nullable), count (integer), unit ("unit" | "kg" | "liter" | "ml" | "gram" | "unknown"), confidence 0–1, depthUncertain boolean, isEmpty boolean (spike bonus — flag empty containers).
  • "Do not invent products you cannot see. Omit anything you are unsure exists."

Counts are estimates — confirmed by the user

Every returned count is a best-effort estimate. The UI presents them as editable defaults, flags depthUncertain rows, and only what the user confirms is written to the baseline. No code path treats a model count as authoritative.

Auth / Infra Decision (corrected)

Detection uses the Gemini Developer API, NOT Vertex. The module already does genkit({ plugins: [googleAI({ apiKey: process.env.GOOGLE_GENAI_API_KEY })], model: googleAI.model("gemini-2.5-flash") }). Keep this. Do NOT switch to vertexAI and do NOT use defineSecret. The key lives in functions/.env.jsdev-stores-prod AND Secret Manager (GOOGLE_GENAI_API_KEY).
KNOWN PRODUCTION RISK — the free Developer API is flaky. It returns HTTP 503 "model is overloaded / high demand" on roughly 2 of 3 calls. The current code has no retry and only catches 429 (not 503), so a 503 currently surfaces as a generic internal error. Before this ships to production, the analyze path MUST gain: (1) retry with backoff (e.g. 2–3 attempts, jittered) around the Gemini call inside reconcilePlaceInventory; and (2) 503 → friendly message handling in the callable (return a "vision temporarily busy, try again" warning, not a hard throw) — mirroring the existing 429 branch. Alternatively, reconsider Vertex (more reliable; the chatbot uses it) as a follow-up if 503s persist after retry. This is the single biggest delivery risk and is called out again in Risks.
  • process.env.GOOGLE_GENAI_API_KEY must be .trim()-ed at read (guard against a trailing newline corrupting the key).
  • Secret verification: firebase functions:secrets:access GOOGLE_GENAI_API_KEY | od -c must end in exactly one \n; set via printf '%s' "key" | firebase functions:secrets:set GOOGLE_GENAI_API_KEY --data-file=- (never echo).

Security Invariants

Authentication

All four callables require a signed-in, non-anonymous user (mirror the spike's detectFridgeProducts guard: reject absent auth and sign_in_provider === "anonymous"). No place data is created or read for anonymous users.

Authorization — ORG MEMBERSHIP (the central new control)

companyId/storeId come from auth.token, NEVER client input. Every callable MUST verify the caller belongs to the target organization via checkOrgMembership: read the caller's own profiles/{auth.uid} (tenant-scoped) and assert organizationIds.includes(organizationId) (falling back to the deprecated organizationId field). For analyzePlacePhotos/confirmPlaceBaseline the organizationId is taken from the loaded place, not the request, then membership-checked — so a member of org A can never read/write org B's places. There is no existing pattern for this (the only org-scoped callable today is admin-only), so it must be built and tested here.

Input validation & trust boundary

Validate every request with the Zod contracts. imageRefs max 5; each path MUST be under the caller's own {companyId}/{storeId}/storage-place-photos/{placeId}/ prefix — reject foreign Storage paths so a caller cannot feed another tenant's/place's images into Gemini. count/fullQty-style numbers are non-negative integers. name non-empty; type within the enum.

Data boundaries & PII

Baseline photos are PII-adjacent: store ONLY under the tenant-scoped Storage path; never log image bytes/URLs or detection text. The fridge-scans log holds counts/model/cost only. Diff-scan photos (Flow B, later) are deleted after the diff; baseline photos are intentionally retained as the place reference. Structured logs (firebase-functions/v2 logger) carry ids only.

Trust escalation — counts are estimates, not authority

No code path treats a Gemini count as authoritative — only the user-confirmed items on confirmPlaceBaseline are persisted as the baseline. (When the later cart flow exists, prices/SKUs MUST be recomputed server-side from the store, never trusted from the client — same rule as the existing cart.)

External service / secret

Reuses the Gemini Developer API already wired in the module (GOOGLE_GENAI_API_KEY in functions/.env.jsdev-stores-prod + Secret Manager). No NEW external service. .trim() the env read. Known reliability risk (503s) is handled with retry + friendly fallback — see Auth/Infra and Risks. Quota/429 AND overload/503 must both return a friendly message, never an unhandled throw.

Risks & Open Questions

High

Gemini Developer API reliability (HTTP 503). The free Developer API returns 503 "overloaded / high demand" on ~2 of 3 calls; current code has no retry and only catches 429. Mitigation (required before prod): retry/backoff around the Gemini call + 503→friendly warning in the callable; consider Vertex as a follow-up if 503s persist. This is the #1 delivery risk — see Auth/Infra.

High

No cart without SKU matching. The baseline stores label + count; turning "missing" into a real cart needs each label mapped to a store SKU via Algolia (tenant-filtered) — UNTESTED. Until that spike succeeds, the feature delivers inventory baselines and a (later) restock list, not a purchasable cart. Deliberately deferred; flagged so monetization expectations are set correctly.

High

Depth / occlusion counting is unreliable (spike-proven). Single-image counts over/under-count behind-items and depthUncertain fires inconsistently. Mitigation: multi-angle (up to 5) merge as the real depth signal + always-editable review + "estimate you confirm" framing. The plan never promises exact counts.

Medium

New org-member authorization path. No existing pattern verifies a non-admin member owns an organizationId (the only org-scoped callable today is admin-only). The new checkOrgMembership is the single guard standing between org A and org B's data — it must be applied on ALL four callables and unit/integration-tested for the cross-org-denied case. A miss here is a cross-tenant data leak.

Medium

Multi-angle de-dup quality. Approach #1 asks one Gemini prompt to de-dup the same item across angles; it may still double-count or merge two distinct items. Mitigation: the review grid is fully editable, and the reconcilePlaceInventory boundary lets a better strategy (per-image detect + server reconcile) drop in later without upstream changes.

Medium

Hebrew identification breadth. Spike validated several real Israeli brands, but the full catalog is large; some products will be mislabeled. Lower stakes than the consumer plan because nothing auto-buys — the label feeds a human-confirmed baseline, and SKU matching (later) is where label quality really bites.

Low

Multi-angle media input shape. Passing up to 5 images as media parts in one ai.generate is supported, but confirm the per-part shape (download URL vs inline base64) against the installed genkit/@genkit-ai/google-genai version during B3 (the spike already does one image; 5 is the same path).

Open

Image cap is 5 per place (confirmed in the brief). Confirm there is no per-org rate limit needed for v1 given cost-per-call is tiny (spike) — plan assumes cost logging only, no hard limiter. Acceptable?

Open

Entry-point placement for the org-gated "Storage places" area: a header link when an org is active, an item in the profile/account area, or a dedicated nav entry? Plan assumes an org-gated link near the account/org area.

Open

Can any org member edit/delete any place in the org, or only createdBy / admins? Plan assumes any member (places are shared org resources); createdBy is audit-only. Confirm.

Open

Re-confirming a baseline (re-running setup on a place that's already confirmed) — overwrite the baseline outright, or keep history? Plan assumes overwrite (simplest); revisit if audit/history is wanted.
