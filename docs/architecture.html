Storebrix Architecture

A production-ready, modular B2B e-commerce platform. Inspired by Shopify, Medusa, Vendure.

Introduction

This document describes the target architecture for Storebrix — a modular monolith with clear boundaries between 5 layers, event-driven communication between modules, and an extensibility layer that lets merchants customize without forking the core.

Core Principle

Don't fork your codebase for clients. Build hooks, events, and webhooks. That's the one decision that separates a product from a consulting project.

🚫 Zero Breaking Changes Constraint

Production store is live. This refactor must not break anything. Every section below is tagged:

SAFE additive only — do now. No data changes, no API contract changes.
DEFERRED valuable but needs design — do later, in a dedicated phase.
BREAKING requires data migration or contract change — do NOT do now.

Start with SAFE only. BREAKING items stay in the plan so we know where we're going.

Current Firestore Structure (preserved as-is)

The refactor does not rename, move, or restructure any Firestore paths. Current layout stays:

  • COMPANIES, STORES — admin/system config (uppercase, admin-only writes)
  • {companyId}/{storeId}/{collection} — all client-writable data (products, orders, cart, clients, profiles, organizations, invoices, suppliers, budgetAccounts, etc.)
  • STORES/{storeId}/private/data — HYP + ezCount credentials

Tenancy: 1 Firebase project, 1 Firebase Auth tenant per store. Store is resolved from incoming domain on the client, passed to backend as storeId. Backend validates the domain→store mapping (hardening this is a SAFE improvement).

Event Bus (Firestore-backed, Outbox Pattern) SAFE

New events/ collection + new triggers. Does not touch existing data. Existing code keeps working; modules opt in one by one.

Platform platform/eventBus/emit.ts 
import { z } from "zod";

// types.ts — single source of truth for schema + type
export const EventSource = z.enum([
  "orders", "payments", "fulfillment", "catalog",
  "cart", "budget", "customers", "webhooks",
]);
export type EventSource = z.infer<typeof EventSource>;

export const StoredEventSchema = z.object({
  id: z.string().min(1),
  type: z.string().regex(/^[a-z_]+\.[a-z_]+$/, {
    message: "type must be 'module.action' (lowercase, dot-separated)",
  }),
  payload: z.unknown(),
  companyId: z.string().min(1).refine(s => !s.includes("/"), {
    message: "companyId must not contain '/'",
  }),
  storeId: z.string().min(1).refine(s => !s.includes("/"), {
    message: "storeId must not contain '/'",
  }),
  createdAt: z.number().int().positive(),
  correlationId: z.string().optional(),
  actorId: z.string().optional(),
  source: EventSource.optional(),
});

export type StoredEvent<T = unknown> =
  Omit<z.infer<typeof StoredEventSchema>, "payload"> & { payload: T };

// emit.ts — validate via zod, no manual guards
export function emit<T = unknown>(
  tx: FirebaseFirestore.Transaction,
  event: Omit<StoredEvent<T>, "id" | "createdAt">
): StoredEvent<T> {
  const ref = db.collection(
    `${event.companyId}/${event.storeId}/events`
  ).doc();
  const stored = {
    ...event, id: ref.id, createdAt: Date.now(),
    correlationId: event.correlationId ?? ref.id,
  };
  // Validate the full envelope before writing. Throws ZodError on invalid input.
  StoredEventSchema.parse(stored);
  tx.set(ref, stored);
  return stored;
}
Platform platform/eventBus/subscribe.ts 
import { onDocumentCreated } from
  "firebase-functions/v2/firestore";
import { StoredEventSchema } from "./types";

export function subscribe<T>(options, handler) {
  return onDocumentCreated(
    { document: "{companyId}/{storeId}/events/{id}", retry: true },
    async (firestoreEvent) => {
      const { companyId, storeId, id: eventId } = firestoreEvent.params;
      const raw = firestoreEvent.data?.data();
      if (!raw) return;

      // Validate full envelope first
      const envelopeResult = StoredEventSchema.safeParse(raw);
      if (!envelopeResult.success) {
        logger.error("eventBus.subscriber.envelopeInvalid", {
          subscriber: options.name, eventId, companyId, storeId,
          issues: envelopeResult.error.issues,
        });
        return;
      }

      const envelope = envelopeResult.data;
      if (envelope.type !== options.type) return;

      // Then validate the payload against the subscriber's schema
      const payloadResult = options.payloadSchema.safeParse(envelope.payload);
      if (!payloadResult.success) {
        logger.error("eventBus.subscriber.payloadInvalid", {
          subscriber: options.name, eventId, companyId, storeId,
          issues: payloadResult.error.issues,
        });
        return;
      }

      await handler(
        { ...envelope, payload: payloadResult.data },
        { companyId, storeId, eventId },
      );
    }
  );
}
Why this pattern?

Outbox pattern: writing the event in the same transaction as the state change guarantees consistency. Either both succeed, or both fail. Firestore triggers then fan out to subscribers with automatic retries.

⚠️ Subscribers must be idempotent (at-least-once delivery)

Firestore triggers retry on failure. Your subscriber may run 2+ times for the same event. Rule: every subscriber handler must be safe to re-run. The platform does NOT ship idempotency helpers — handlers design their own natural keys (e.g. delivery note keyed by orderId) or their own per-domain markers. Pick the pattern that fits what you're writing.

Pattern A — natural-key create (for creates):

async function createDeliveryNote(orderId, companyId, storeId) {
  const ref = db.doc(`${companyId}/${storeId}/delivery_notes/${orderId}`);
  try {
    await ref.create({ orderId, createdAt: Date.now() });
  } catch (err) {
    if (err.code === 6) return;   // ALREADY_EXISTS — already processed
    throw err;
  }
}

Pattern B — event-id guard (handler's own responsibility — for counters / shared aggregates):

await db.runTransaction(async (tx) => {
  const markerRef = db.doc(
    `${companyId}/${storeId}/idempotency_stats/${eventId}`
  );
  const applied = await tx.get(markerRef);
  if (applied.exists) return;

  // ... update shared counter ...
  tx.set(markerRef, { at: Date.now() });
});

In Pattern B the handler creates its own marker collection under its store path — the platform does not dictate where.

🔗 correlationId — tracing a causal chain

Every event carries a correlationId. The platform auto-fills it with the event's own id when the caller doesn't provide one. That first event becomes the root of the chain.

Downstream subscribers must propagate it. When a subscriber emits a new event in reaction, it copies correlationId from the incoming event:

const downstream = emit(tx, {
  type: "fulfillment.delivery_note_created",
  payload: { ... },
  companyId, storeId,
  correlationId: event.correlationId,  // propagate
  source: "fulfillment",
});
// downstream is the full StoredEvent — downstream.id, downstream.correlationId, etc.

Query a full trace with where("correlationId", "==", X). The root event within the results is the one where id === correlationId.

Audit Log SAFE

New collection. Business record of every meaningful action, shown to merchants in the admin UI (Order History tab, etc.). Not GCP Cloud Logging — that's for engineering debugging, this is the product-level audit trail with long retention.

Stores a diff only (via microdiff) — not full before/after — to keep docs small and reduce PII exposure.

Platform platform/audit/record.ts 
import admin from "firebase-admin";
import diff from "microdiff";
import { AuditEntry, AuditEntrySchema, TargetType } from "./types";

export function record(
  tx: FirebaseFirestore.Transaction,
  params: {
    eventType: string;
    actorId: string;
    targetType: TargetType;
    targetId: string;
    before?: unknown;
    after?: unknown;
    summary?: string;
    companyId: string;
    storeId: string;
    metadata?: { ip?: string; userAgent?: string };
    correlationId?: string;
  },
): FirebaseFirestore.DocumentReference {
  const db = admin.firestore();
  const path = `${params.companyId}/${params.storeId}/audit_log`;
  const ref = db.collection(path).doc();

  const entry: AuditEntry = {
    id: ref.id,
    eventType: params.eventType,
    actorId: params.actorId,
    targetType: params.targetType,
    targetId: params.targetId,
    diff: params.before !== undefined && params.after !== undefined
      ? diff(params.before as Record<string, unknown>, params.after as Record<string, unknown>)
      : undefined,
    summary: params.summary,
    companyId: params.companyId,
    storeId: params.storeId,
    createdAt: Date.now(),
    metadata: params.metadata,
    correlationId: params.correlationId,
  };

  AuditEntrySchema.parse(entry);

  tx.set(ref, entry);
  return ref;
}

Backend modules/cart/internal/cartService.ts 
import { catalogModule } from "modules/catalog";
import { pricingModule } from "modules/pricing";
import { emit } from "platform/eventBus";

export const cartService = {
  async addItem({ cartId, productId, variantId, quantity }, ctx) {
    // Call other module's PUBLIC API
    const product = await catalogModule.findProduct(productId, ctx);
    if (!product) throw new Error("Product not found");

    return db.runTransaction(async (tx) => {
      const cart = await getCart(tx, cartId);
      cart.items.push({ productId, variantId, quantity });

      // Recalculate via pricing module
      cart.totals = await pricingModule.calculateCart(cart, ctx);

      tx.update(db.doc(`carts/${cartId}`), cart);

      // Outbox: event in same transaction
      emit(tx, {
        type: "cart.item_added",
        payload: { cartId, productId, variantId, quantity },
        tenantId: ctx.tenantId,
      });

      return cart;
    });
  },
};

📋 Module: Orders Layer 2 SAFE

Owns: Orders, state machine. Uses: cart, payments, catalog. Published events: order.placed, order.fulfilled, order.cancelled.

Keep single status field for now

Two-level state (order status + per-line-item status) is the right model for B2B (partial shipments, partial refunds), but migrating existing orders is a BREAKING data change. Stay with the current linear state for now. Revisit when a real use case forces it.

State Machine DEFERRED

The strict transition table below is the target. Enforcing it on existing orders would reject any order in a state not listed — which could break production data. Land the transitions table as a read-only helper first (reject invalid new transitions, don't retroactively validate existing orders), then enforce fully once data is audited.

draft ──> pending ──> paid ──> processing ──> in_delivery ──> delivered ──> completed │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ cancelled cancelled cancelled cancelled refunded
Backend modules/orders/internal/stateMachine.ts 
const ALLOWED_TRANSITIONS = {
  draft: ["pending", "cancelled"],
  pending: ["paid", "cancelled"],
  paid: ["processing", "cancelled", "refunded"],
  processing: ["in_delivery", "cancelled"],
  in_delivery: ["delivered"],
  delivered: ["completed", "refunded"],
  completed: ["refunded"],
  cancelled: [],
  refunded: [],
};

export const orderStateMachine = {
  async transition(orderId, newStatus, ctx) {
    return db.runTransaction(async (tx) => {
      const order = await getOrder(tx, orderId);

      const allowed = ALLOWED_TRANSITIONS[order.status];
      if (!allowed.includes(newStatus)) {
        throw new Error(
          `Invalid transition: ${order.status} → ${newStatus}`
        );
      }

      tx.update(orderRef, {
        status: newStatus,
        statusHistory: FieldValue.arrayUnion({
          from: order.status, to: newStatus, at: Date.now(),
        }),
      });

      emit(tx, {
        type: `order.${newStatus}`,
        payload: { orderId, previousStatus: order.status },
        tenantId: ctx.tenantId,
      });
    });
  },
};

🪝 Webhooks Layer 3 Extensibility SAFE

Let merchants subscribe to your events via HTTP. The killer feature that makes you a platform.

Integration integration/webhooks/dispatcher.ts 
import crypto from "crypto";
import { subscribe } from "platform/eventBus";
import { getSubscriptions } from "./subscriptionService";

// ONE subscriber that listens to ALL events
export const webhookDispatcher = subscribe(async (event) => {
  const subs = await getSubscriptions(event.tenantId, event.type);

  for (const sub of subs) {
    try {
      await dispatch(sub, event);
    } catch (err) {
      await markFailedAttempt(sub.id, event.id, err);
    }
  }
});

async function dispatch(sub, event) {
  const body = JSON.stringify(event);
  const signature = crypto
    .createHmac("sha256", sub.secret)
    .update(body)
    .digest("hex");

  const res = await fetch(sub.url, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "X-Storebrix-Signature": signature,
      "X-Storebrix-Event": event.type,
      "X-Storebrix-Event-Id": event.id,
    },
    body,
    signal: AbortSignal.timeout(10_000),
  });

  if (!res.ok) throw new Error(`status ${res.status}`);
}

🔌 Plugins Layer 3 OUT OF SCOPE

Third-party / merchant-authored plugins are not in scope for this refactor.

Why out of scope

Running untrusted code inside our Cloud Functions is a serious security problem — secrets exposure, crashes, network exfiltration, resource exhaustion. Doing it safely requires either a separate runtime (Cloud Run per plugin, V8 isolates, Workers) or a review/signing process. That's a dedicated project, not part of this refactor.

For now: any custom behavior is first-party code that lives inside the modules. Extensibility is provided via webhooks (external systems subscribe via HTTP) and hook points (used only by first-party code).

🎯 Hook Points Layer 3 SAFE

Synchronous extension points in core flows. Used by first-party modules to compose behavior cleanly (plugin authoring is out of scope — see above).

Cold-start reload (Firebase Cloud Functions)

Cloud Functions run multiple instances. An in-memory registry map is per-process, not shared. Every cold start must re-register the same handlers deterministically — otherwise behavior varies by which instance serves the request.

// platform/bootstrap.ts — runs at module load on every cold start
import { registerOrderHooks } from "modules/orders/hooks";
import { registerPricingHooks } from "modules/pricing/hooks";
registerOrderHooks();
registerPricingHooks();
// same imports, same calls, every instance → deterministic
Integration integration/hooks/registry.ts 
type HookHandler<I, O> = (input: I, ctx: Ctx) => Promise<O>;

const registry = new Map<string, HookHandler<any, any>[]>();

export function registerHook<I, O>(
  name: string,
  handler: HookHandler<I, O>
) {
  const existing = registry.get(name) ?? [];
  registry.set(name, [...existing, handler]);
}

export async function runHook<I>(
  name: string,
  input: I,
  ctx: Ctx
): Promise<I> {
  const handlers = registry.get(name) ?? [];
  let result = input;

  for (const h of handlers) {
    result = await h(result, ctx);
  }

  return result;
}
Backend modules/orders/internal/orderService.ts 
import { runHook } from "integration/hooks";

export const orderService = {
  async create(input, ctx) {
    // ← Plugins can validate/modify input here
    input = await runHook("order.before_create", input, ctx);

    // ← Plugins can inject custom shipping logic
    const shipping = await runHook("order.calculate_shipping", input, ctx);

    // Core logic runs between hooks
    const order = await orderRepo.create({ ...input, shipping });

    // ← Plugins can react after creation
    await runHook("order.after_create", order, ctx);

    // Fire async event for subscribers
    await emit({ type: "order.placed", payload: order });

    return order;
  },
};

Migration Roadmap

Start safe. Land everything additive first. Anything marked BREAKING is deferred until we have a dedicated migration plan and backup.

Stage 0 — Non-Breaking Foundation SAFE

Entirely additive. Can ship incrementally to production without risk.

  • Code reorganization — move files into platform/, modules/, integration/ layout. Paths in Firestore stay identical. No runtime change.
  • Public index.ts per module — wrap existing services, expose only the public API.
  • ESLint boundaries — enforce layer import rules; raw firebase-admin/firestore banned outside platform/db.
  • tenantDb(ctx) typed wrapper — new helper that makes tenantless queries impossible. Existing call sites migrate gradually.
  • Event bus — new events/ collection + emit helper + subscribe. Outbox pattern.
  • Subscriber idempotency conventions — natural-key writes + event-id guards for shared counters (see Event Bus callout).
  • Audit log — new audit_log collection, diff-based via microdiff.
  • Webhooks — new collection webhookSubscription + signed HTTP dispatcher.
  • Hook registry — in-memory, with cold-start re-register on every instance.
  • Idempotency keys on payment + order creation (already landed for delivery notes).
  • Harden domain → store resolution — backend re-verifies the incoming domain against the store's allowed urls; client-provided storeId is only trusted if it matches. Current client-side flow keeps working.
  • Fix two known structure bugs in todo.md (profile vs profiles, org actions path) — both BREAKING, handle in Stage 2.

Stage 1 — Wire Up Cross-Module Flows SAFE

Use the foundation to decouple what's currently tangled.

  • First 5 events: order.placed, order.paid, payment.completed, delivery_note.created, invoice.created.
  • Move budget/fulfillment cross-talk onto the event bus (currently direct function calls).
  • State machine helper as read-only guard for new transitions only — existing orders unaffected.
  • Feature flags (simple Firestore config map per store).
  • Rate limiting on public callables.

Stage 2 — Data Migrations BREAKING

Each item requires a backup + migration script + rollback plan. Do one at a time, verify, then next.

  • Unify profile → profiles (rewrite createCompany.ts; migrate any existing docs).
  • Move organizations/{orgId}/actions under {companyId}/{storeId}/organizations/{orgId}/actions; update security rules.
  • Full state machine enforcement (after auditing existing order states).
  • Remove deprecated organizationId (singular) from Profile in favor of organizationIds.

Stage 3 — Fill Commerce Gaps (driven by client needs) SAFE

  • Product variants (new sub-entities, additive).
  • Tiered/volume pricing (new pricing rules).
  • Payment provider abstraction (when 2nd gateway needed).
  • Shipping zones + rates.
  • Returns / RMA flow.
  • Promo codes.

Stage 4 — Scale (later) DEFERRED

  • Multi-currency + multi-language.
  • Two-level order state (order + per-line-item) — requires data migration.
  • Theme system for storefronts.
  • Admin dashboard with real KPIs.
  • RBAC for admin users.
  • Comprehensive test suite (out of scope for current refactor per decision).

Modular monolith = you keep the simplicity of one codebase/one DB, but enforce Shopify-level boundaries between parts of the system — so your code ages well instead of turning into a mud ball.
