Storebrix Architecture

A production-ready, modular B2B e-commerce platform. Inspired by Shopify, Medusa, Vendure.

Introduction

This document describes the target architecture for Storebrix — a modular monolith with clear boundaries between 5 layers, event-driven communication between modules, and an extensibility layer that lets merchants customize without forking the core.

Core Principle

Don't fork your codebase for clients. Build hooks, events, and webhooks. That's the one decision that separates a product from a consulting project.

Architecture Rules

The golden rules. Enforce these with ESLint no-restricted-imports.

Layer CAN import from CANNOT import from
5. Storefront client-sdk, shared ui-kit Any backend code directly
4. Admin client-sdk, shared ui-kit Any backend code directly
3. Integration platform/*, modules/*/index.ts (public API) modules/*/internal/*
2. Modules platform/*, other modules/*/index.ts Other modules' internals, integration/*
1. Platform Nothing (it's the base) Anything above
If you break these rules...

You get the "flat monolith" mess — changes in one place break unrelated things, onboarding devs is painful, and every client feature forks the core. These rules exist because every production platform learned them the hard way.

The foundation. Nobody in business logic touches the database directly or implements their own auth. Everyone uses platform/*.

Event Bus (Firestore-backed, Outbox Pattern)

Platform platform/eventBus/emit.ts 
import { FieldValue } from "firebase-admin/firestore";
import { db } from "../db";

export interface DomainEvent<T = any> {
  type: string;
  payload: T;
  tenantId: string;
  correlationId?: string;
  version?: number;
}

/** Emit an event in the SAME transaction as state writes (outbox pattern) */
export function emitInTransaction(
  tx: FirebaseFirestore.Transaction,
  event: DomainEvent
): void {
  const ref = db.collection("events").doc();
  tx.set(ref, {
    id: ref.id,
    ...event,
    createdAt: Date.now(),
    traceId: event.correlationId ?? ref.id,
    version: event.version ?? 1,
  });
}

/** Standalone emit (NOT transactional — use sparingly) */
export async function emit(event: DomainEvent) {
  await db.collection("events").add({
    ...event,
    createdAt: Date.now(),
  });
}
Platform platform/eventBus/subscribe.ts 
import { onDocumentCreated } from
  "firebase-functions/v2/firestore";

/**
 * Creates a Firebase trigger that subscribes to events.
 * Each module calls this to register its subscribers.
 */
export function createEventSubscriber<T>(
  handler: (event: DomainEvent<T>) => Promise<void>
) {
  return onDocumentCreated(
    { document: "events/{id}", retry: true },
    async (event) => {
      const data = event.data?.data() as DomainEvent<T>;
      if (!data) return;

      try {
        await handler(data);
      } catch (err) {
        logger.error("event handler failed", {
          eventId: event.params.id,
          type: data.type,
          err,
        });
        throw err; // triggers retry
      }
    }
  );
}
Why this pattern?

Outbox pattern: writing the event in the same transaction as the state change guarantees consistency. Either both succeed, or both fail. Firestore triggers then fan out to subscribers with automatic retries.

Audit Log

Platform platform/audit/auditLog.ts 
export interface AuditEntry {
  action: string;          // "order.status_changed"
  actorId: string;         // user id or "system"
  targetType: string;      // "order"
  targetId: string;        // the order id
  before?: any;             // state before change
  after?: any;              // state after change
  tenantId: string;
  ip?: string;
  userAgent?: string;
}

export const auditLog = {
  async record(entry: AuditEntry): Promise<void> {
    await db.collection("audit_log").add({
      ...entry,
      at: Date.now(),
    });
  },

  async queryByTarget(type: string, id: string) {
    return db.collection("audit_log")
      .where("targetType", "==", type)
      .where("targetId", "==", id)
      .orderBy("at", "desc")
      .get();
  },
};

Backend modules/cart/internal/cartService.ts 
import { catalogModule } from "modules/catalog";
import { pricingModule } from "modules/pricing";
import { emitInTransaction } from "platform/eventBus";

export const cartService = {
  async addItem({ cartId, productId, variantId, quantity }, ctx) {
    // Call other module's PUBLIC API
    const product = await catalogModule.findProduct(productId, ctx);
    if (!product) throw new Error("Product not found");

    return db.runTransaction(async (tx) => {
      const cart = await getCart(tx, cartId);
      cart.items.push({ productId, variantId, quantity });

      // Recalculate via pricing module
      cart.totals = await pricingModule.calculateCart(cart, ctx);

      tx.update(db.doc(`carts/${cartId}`), cart);

      // Outbox: event in same transaction
      emitInTransaction(tx, {
        type: "cart.item_added",
        payload: { cartId, productId, variantId, quantity },
        tenantId: ctx.tenantId,
      });

      return cart;
    });
  },
};

📋 Module: Orders Layer 2

Owns: Orders, state machine. Uses: cart, payments, catalog. Published events: order.placed, order.fulfilled, order.cancelled.

State Machine

draft ──> pending ──> paid ──> processing ──> in_delivery ──> delivered ──> completed │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ cancelled cancelled cancelled cancelled refunded
Backend modules/orders/internal/stateMachine.ts 
const ALLOWED_TRANSITIONS = {
  draft: ["pending", "cancelled"],
  pending: ["paid", "cancelled"],
  paid: ["processing", "cancelled", "refunded"],
  processing: ["in_delivery", "cancelled"],
  in_delivery: ["delivered"],
  delivered: ["completed", "refunded"],
  completed: ["refunded"],
  cancelled: [],
  refunded: [],
};

export const orderStateMachine = {
  async transition(orderId, newStatus, ctx) {
    return db.runTransaction(async (tx) => {
      const order = await getOrder(tx, orderId);

      const allowed = ALLOWED_TRANSITIONS[order.status];
      if (!allowed.includes(newStatus)) {
        throw new Error(
          `Invalid transition: ${order.status} → ${newStatus}`
        );
      }

      tx.update(orderRef, {
        status: newStatus,
        statusHistory: FieldValue.arrayUnion({
          from: order.status, to: newStatus, at: Date.now(),
        }),
      });

      emitInTransaction(tx, {
        type: `order.${newStatus}`,
        payload: { orderId, previousStatus: order.status },
        tenantId: ctx.tenantId,
      });
    });
  },
};

💰 Module: Budget Layer 2 B2B

B2B-specific. Tracks debt/credit per organization. Pure event-driven — listens to fulfillment & payment events.

Backend modules/budget/subscribers.ts 
import { createEventSubscriber } from "platform/eventBus";
import { budgetService } from "./internal/budgetService";

// Debt is added when delivery note is issued
export const onDeliveryNote = createEventSubscriber(async (event) => {
  if (event.type !== "fulfillment.delivery_note.created") return;

  const { organizationId, total, deliveryNoteId } = event.payload;
  if (!organizationId) return;

  await budgetService.addTransaction({
    organizationId,
    type: "delivery_note",
    amount: total,
    deliveryNoteId,
  });
});

// Credit when payment completes
export const onPaymentReceived = createEventSubscriber(async (event) => {
  if (event.type !== "payment.captured") return;

  await budgetService.addTransaction({
    organizationId: event.payload.organizationId,
    type: "payment_received",
    amount: -event.payload.amount, // reduces debt
  });
});

Notice the decoupling

🪝 Webhooks Layer 3 Extensibility

Let merchants subscribe to your events via HTTP. The killer feature that makes you a platform.

Integration integration/webhooks/dispatcher.ts 
import crypto from "crypto";
import { createEventSubscriber } from "platform/eventBus";
import { getSubscriptions } from "./subscriptionService";

// ONE subscriber that listens to ALL events
export const webhookDispatcher = createEventSubscriber(async (event) => {
  const subs = await getSubscriptions(event.tenantId, event.type);

  for (const sub of subs) {
    try {
      await dispatch(sub, event);
    } catch (err) {
      await markFailedAttempt(sub.id, event.id, err);
    }
  }
});

async function dispatch(sub, event) {
  const body = JSON.stringify(event);
  const signature = crypto
    .createHmac("sha256", sub.secret)
    .update(body)
    .digest("hex");

  const res = await fetch(sub.url, {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "X-Storebrix-Signature": signature,
      "X-Storebrix-Event": event.type,
      "X-Storebrix-Event-Id": event.id,
    },
    body,
    signal: AbortSignal.timeout(10_000),
  });

  if (!res.ok) throw new Error(`status ${res.status}`);
}

🎯 Hook Points Layer 3

Synchronous extension points in your core flows. Plugins register handlers that run at these moments.

Integration integration/hooks/registry.ts 
type HookHandler<I, O> = (input: I, ctx: Ctx) => Promise<O>;

const registry = new Map<string, HookHandler<any, any>[]>();

export function registerHook<I, O>(
  name: string,
  handler: HookHandler<I, O>
) {
  const existing = registry.get(name) ?? [];
  registry.set(name, [...existing, handler]);
}

export async function runHook<I>(
  name: string,
  input: I,
  ctx: Ctx
): Promise<I> {
  const handlers = registry.get(name) ?? [];
  let result = input;

  for (const h of handlers) {
    result = await h(result, ctx);
  }

  return result;
}
Backend modules/orders/internal/orderService.ts 
import { runHook } from "integration/hooks";

export const orderService = {
  async create(input, ctx) {
    // ← Plugins can validate/modify input here
    input = await runHook("order.before_create", input, ctx);

    // ← Plugins can inject custom shipping logic
    const shipping = await runHook("order.calculate_shipping", input, ctx);

    // Core logic runs between hooks
    const order = await orderRepo.create({ ...input, shipping });

    // ← Plugins can react after creation
    await runHook("order.after_create", order, ctx);

    // Fire async event for subscribers
    await emit({ type: "order.placed", payload: order });

    return order;
  },
};

Migration Roadmap

You don't need to rewrite. Do this incrementally.

Stage 1 — Stop the Bleeding (1-2 months)

Production safety before adding features:

  • Idempotency keys on payment + order creation
  • Stock reservation during checkout
  • Append-only audit log for orders
  • Rate limiting on public APIs
  • Feature flags (simple Firestore config map)

Stage 2 — Unlock Modularity (2-3 months) ⭐ THE KEY STAGE

This is what separates "custom build for every client" from "platform":

  • Event bus (Firestore-based, outbox pattern)
  • Webhooks — merchants subscribe to events via signed HTTP POST
  • Hook points in the order flow
  • Per-store config schema
  • Plugin loader — bundle custom code per store

Concrete first move:

  1. Create domainEvents collection + emitEvent(type, payload) helper
  2. Pick 5 key events: order.placed, order.fulfilled, payment.completed, delivery_note.created, invoice.created
  3. Build webhookSubscription collection
  4. Dispatcher sends signed POSTs on each event
  5. → Now every future feature plugs in via events instead of forking core

Stage 3 — Fill Commerce Gaps (3-6 months)

Driven by actual client needs:

  • Product variants (if apparel/multi-option needed)
  • Tiered/volume pricing (essential for B2B wholesale)
  • Payment provider abstraction (when 2nd client wants different gateway)
  • Shipping zones + rates
  • Returns/RMA flow
  • Promo codes

Stage 4 — Scale (6-12 months)

  • Multi-currency + multi-language
  • Migrate "stores" → "channels" as tenancy primitive
  • Theme system for storefronts
  • Admin dashboard with real KPIs
  • Comprehensive test suite
  • RBAC for admin users
The one-line summary

Modular monolith = you keep the simplicity of one codebase/one DB, but enforce Shopify-level boundaries between parts of the system — so your code ages well instead of turning into a mud ball.
