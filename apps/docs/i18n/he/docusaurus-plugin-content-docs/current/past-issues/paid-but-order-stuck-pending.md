# ההזמנה חויבה בסליקה אך נשארה במצב "ממתין"

| | |

| **נמצאו עד כה** | **7 הזמנות · ₪12,569.39** |

## בשורה אחת

תשלום בכרטיס אשראי **מצליח** בסליקה (הלקוח אף מקבל קבלה), אבל ההזמנה ממשיכה
להופיע כ**לא משולמת** (`pending_j5`) — כי נתיב הקוד שרשם את התשלום מעולם לא עדכן
את ההזמנה שהיא שולמה.

> שונה מ[בעיית הטיוטה הכפולה](./order-looked-unpaid-but-was-paid.md): שם הכסף ישב
> על הזמנה *תאומה*; כאן הכסף נמצא על *ההזמנה הזו*, אבל הסטטוס שלה מעולם לא התעדכן.

## התסמין

עבור כל הזמנה מושפעת:
- הסליקה מציגה את החיוב כ**מאושר** (קבלה הופקה), אבל
- ההזמנה תקועה ב-`status: completed/delivered` + **`paymentStatus: pending_j5`**, וכן
- אין עבורה **שום שורת עסקה בספר החשבונות**.

## סיבת השורש (התגלית האמיתית)

רישום תוצאת התשלום נמצא **באמצע מעבר בין שני מקומות**:

- **ישן (לקוח / דפדפן):** אחרי ההפניה מהסליקה, הדפדפן כותב ישירות את ההזמנה +
  מסמך התשלום. הוא מגדיר `paymentStatus: "pending_j5"` ושומר את רשומת התשלום, אבל
  **לא רושם שום שורת עסקה בספר החשבונות**.
- **חדש (שרת / Cloud Function):** מאמת מחדש את חתימת הסליקה ו**רושם שורת עסקה**;
  ואז מנגנון המנוי `markOrderPaidOnTransactionPosted` מעביר את ההזמנה ל-`completed`.

הקוד ה**יחיד** שמגדיר `paymentStatus: "completed"` הוא אותו מנוי, והוא **רץ רק
כשנרשמת שורת עסקה**. חיובים של תשלום ישיר / לינק-תשלום של אדמין עדיין עוברים
ב**נתיב הלקוח הישן**, שלא רושם שורת עסקה → ההזמנה נתקעת ב-`pending_j5` למרות
שהכרטיס חויב.

(משני, מאותו סוג: רושם ה-J5 בשרת רשם תמיד "אישור" (auth) גם כשהתוצאה הייתה חיוב
אמיתי — מה שגם ממופה ל-`pending_j5`.)

**למה בכלל שני מקומות:** הצוות מעביר את רישום התשלום מהלקוח לשרת (השרת מאמת מחדש
ומהווה מקור אמת יחיד דרך ספר החשבונות). נתיב ה-J5 הועבר; נתיב ה**תשלום הישיר /
לינק-אדמין לא** — "מעבר #2" מעולם לא הושלם, ולכן הכתיבה הישנה מצד הלקוח עדיין פעילה.

## התיקון

**תיקון זמני (נכתב, לא בוצע commit)** — מתקן את סטטוס ההזמנה בשני הנתיבים הפעילים:
- `apps/store/src/appApi/index.ts` (ה-fallback הישן ב-`onOrderPaid`): כשהתוצאה אכן
  חויבה (`CCode === "0"`), להגדיר `paymentStatus: "completed"` במקום `pending_j5`.
- `functions/src/modules/ledger/api/recordHypJ5Auth.ts`: תוצאת `CCode "0"` רושמת
  עסקת `hyp_direct` (→ `completed`) במקום `hyp_j5_auth` (→ `pending_j5`).

**תיקון נכון (עתידי):** להשלים את המעבר — לנתב תשלומים ישירים / לינק-אדמין דרך
ה-handler הקיים בשרת `recordHypDirectPayment` (מאמת + רושם שורת עסקה), ואז למחוק
את הכתיבה הישנה מצד הלקוח. מקום אחד, בלי באג.

## עדיין פתוח / כדאי לשים לב

- ה**תיקון הזמני מתקן רק את סטטוס ההזמנה, לא את ספר החשבונות** — לחיובים האלה עדיין
  אין שורת עסקה, ולכן הם לא יופיעו בדוחות ההכנסות עד שתבוצע השלמה (נדחה).
- התיקון **לא הופץ** — עד שכן, הזמנות ישירות / לינק-אדמין חדשות עדיין עלולות
  להיתקע. סריקה תקופתית של הזמנות `pending_j5` שיש להן חיוב מוצלח (עם קבלה) תאתר חדשות.
